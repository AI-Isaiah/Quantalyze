#### C-0326 · L166 · red-team c9 ⏳
- **Title**: Inverse perp (BTC/USD:BTC) cost is in BASE units — cost/price returns wrong amt_base for ALL inverse trades
- **Summary**: `amt_from_cost = (cost / price) if cost > 0 else amount` (line 166) presumes cost is QUOTE-denominated (i.e. cost = amount_base × price). This is true for linear perps (cost in USDT). Result: every inverse perp trade gets a silently-wrong amt_base, and the position-tracking state drifts to garbage. The current OKX_PERP_CONTRACT_SIZE table contains only linear (USDT-quoted) symbols, so the defensive override never fires (raw_symbol mismatch) and the broken cost/price path always wins for inverse fills.
- **Evidence**: analytics-service/services/equity_reconstruction.py:166 (single cost/price line, no venue/contract-shape gating). Line 703-714 comment confirms ccxt uses BTC/USD:BTC for inverse — yet `is_perp = ':' in raw_symbol` (line 714) puts inverse trades through the same cost/price branch. OKX_PERP_CONTRACT_SIZE (123-138) has ONLY linear keys (USDT:USDT suffix). No inverse-perp regression test exists.
- **Fix**: Detect inverse shape: `is_inverse = raw_symbol.endswith(':' + raw_symbol.split('/')[0])` (the settle ccy equals the base). For inverse: `amt_base = (amount × ctVal × price_inverse_formula)` — or skip inverse entirely and emit `allocator.equity.unsupported_inverse_perp` audit event with a TODO. Add OKX_INVERSE_PERP_CONTRACT_SIZE table OR explicit pre-flight raise so inverse fills don't silently corrupt the replay.
