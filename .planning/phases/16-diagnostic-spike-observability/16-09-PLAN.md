**Step B: Run the PostHog query.** Use either the posthog-node SDK or a one-shot `curl` call against the PostHog HogQL/events endpoint. Two acceptable shapes:

Shape A (HogQL — preferred, exact criteria):
```bash
curl -sS -X POST "${POSTHOG_HOST:-https://us.i.posthog.com}/api/projects/${POSTHOG_PROJECT_ID}/query/" \
  -H "Authorization: Bearer ${POSTHOG_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{
    "query": {
      "kind": "HogQLQuery",
      "query": "SELECT count(*) AS mobile_starts FROM events WHERE event = '"'"'wizard_start'"'"' AND properties.$device_type = '"'"'Mobile'"'"' AND timestamp > now() - INTERVAL 30 DAY"
    }
  }'
```

Shape B (events endpoint with property filter):
```bash
curl -sS -G "${POSTHOG_HOST:-https://us.i.posthog.com}/api/projects/${POSTHOG_PROJECT_ID}/events/" \
  -H "Authorization: Bearer ${POSTHOG_API_KEY}" \
  --data-urlencode "event=wizard_start" \
  --data-urlencode "after=$(date -u -v-30d +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u --date='-30 days' +%Y-%m-%dT%H:%M:%SZ)" \
  --data-urlencode 'properties=[{"key":"$device_type","value":"Mobile","operator":"exact"}]'
```

**Run BOTH:** Shape A for the count, Shape B for spot-check (first 5 events to confirm the criterion is hitting real rows, not zero by accident).

Then run a CROSS-CHECK query for `$viewport_width < 768` (HogQL):
```sql
SELECT count(*) AS small_viewport_starts
FROM events
WHERE event = 'wizard_start'
  AND properties.$viewport_width < 768
  AND timestamp > now() - INTERVAL 30 DAY
```

**If credentials are missing or the API returns 4xx:** capture the error body verbatim, treat the result as `0` (per RESEARCH L1106 — "Plan 9 documents '0' if PostHog unavailable; downstream DESIGN-04 ships 640px gate"), and document the credential issue in the audit doc.
