**FIX 11 (outside-voice review): use EXPLICIT contextvars.Token reset, NOT clear_contextvars()
alone.** The pattern is:

```python
token = correlation_id_var.set(cid)
try:
    response = await call_next(request)
finally:
    correlation_id_var.reset(token)
```

`clear_contextvars()` removes ALL bindings; if any other code in the request scope
(or a sibling middleware) bound an unrelated contextvar, `clear_contextvars()` would
drop that too. Token-based reset surgically removes ONLY the binding we set, eliminating
both bleed risk AND collateral damage.

A 100-request sequential stress test asserts no bleed across requests AND no zombie
contextvar binding at request end.

// Wire form is PascalCase ("X-Correlation-Id") to match existing X-Api-Version /

export const CORRELATION_HEADER = "x-correlation-id";

/**
 * Read inbound correlation_id from the current request headers, or generate a
 * fresh UUID v4 if absent. Use at every Route Handler / Server Action entry
 * that fans out to analyticsRequest() so the chain stays joinable end-to-end.
 *
 * Node 20+ provides `crypto.randomUUID()` as a global; this repo is on Node 20
 * (CI) and Vercel runtime is 24 LTS — no polyfill needed.

export async function getCorrelationId(): Promise<string> {

  return h.get(CORRELATION_HEADER) ?? crypto.randomUUID();

**Step B: Create `analytics-service/services/logging_config.py`** per PATTERNS.md L405-442 + RESEARCH Pattern 2 L389-428, **with EXPLICIT Token-based contextvars reset per FIX 11** (NOT clear_contextvars() alone):

```python
"""Phase 16 / OBSERV-02 + OBSERV-09 — structlog + Sentry + correlation_id ASGI middleware.

Asserted invariants:
  1. configure_logging() is idempotent and called exactly once at process startup.
  2. CorrelationMiddleware binds correlation_id to a structlog contextvar AND a
     Sentry tag for the request scope.
  3. EXPLICIT Token-based reset (FIX 11): we use `correlation_id_var.set(cid)` →
     keep the Token → `correlation_id_var.reset(token)` in finally. clear_contextvars()
     is too broad — it would drop ANY other contextvar binding from siblings or
     downstream code. Token-based reset surgically removes ONLY the binding we set.
  4. Response echoes X-Correlation-Id back so the caller can grep against
     client-side logs.
"""

from __future__ import annotations

from contextvars import ContextVar
from uuid import uuid4

import sentry_sdk
import structlog
from starlette.middleware.base import BaseHTTPMiddleware
from starlette.requests import Request
from starlette.responses import Response


# Module-scope ContextVar for surgical Token-based reset (FIX 11).
# This is the SAME contextvar that structlog's merge_contextvars reads from —
# we register it explicitly so we can hold the Token and reset it precisely.
correlation_id_var: ContextVar[str | None] = ContextVar("correlation_id", default=None)


def configure_logging() -> None:
    """Configure structlog ONCE at process startup. Call before app = FastAPI()."""
    structlog.configure(
        processors=[
            structlog.contextvars.merge_contextvars,
            structlog.processors.add_log_level,
            structlog.processors.TimeStamper(fmt="iso", utc=True),
            structlog.processors.dict_tracebacks,
            structlog.processors.JSONRenderer(sort_keys=True),
        ],
        wrapper_class=structlog.make_filtering_bound_logger(20),  # INFO+
        cache_logger_on_first_use=True,
    )


class CorrelationMiddleware(BaseHTTPMiddleware):
    """ASGI middleware that binds correlation_id for the request scope.

    Header on the wire is `X-Correlation-Id` (PascalCase per Next.js precedent
    in src/lib/analytics-client.ts:70-71). Read lower-case here per HTTP
    normalization.

    FIX 11: Uses explicit Token-based reset instead of clear_contextvars(). The
    Token captures the previous value of correlation_id_var so reset() restores
    it exactly. This protects against:
      - bleed across sequential requests (the headline invariant)
      - collateral damage to other contextvars bound by sibling middleware
      - test brittleness from clear_contextvars wiping unrelated state
    """

    HEADER_NAME = "x-correlation-id"

    async def dispatch(self, request: Request, call_next):  # type: ignore[no-untyped-def]
        cid = request.headers.get(self.HEADER_NAME) or str(uuid4())

        # Capture the Token so we can surgically reset (FIX 11).
        cv_token = correlation_id_var.set(cid)
        # Also bind via structlog's contextvars helper so the merge_contextvars
        # processor picks up the value automatically. Capture its token too if
        # the structlog version exposes one; older versions return None.
        sl_token = structlog.contextvars.bind_contextvars(
            correlation_id=cid,
            method=request.method,
            path=request.url.path,
        )
        with sentry_sdk.new_scope() as scope:
            scope.set_tag("correlation_id", cid)
            try:
                response: Response = await call_next(request)
            finally:
                # Surgical reset (FIX 11) — Token-based, not clear_contextvars().
                # If sl_token is a Mapping (newer structlog), unbind; older versions
                # return None and we fall back to unbind_contextvars by name.
                try:
                    if sl_token is not None:
                        structlog.contextvars.reset_contextvars(**sl_token)
                except (AttributeError, TypeError):
                    structlog.contextvars.unbind_contextvars(
                        "correlation_id", "method", "path"
                    )
                correlation_id_var.reset(cv_token)
        # Echo header back so caller can join client-side and server-side logs.
        response.headers["X-Correlation-Id"] = cid
        return response
```

**Note on `structlog.contextvars.reset_contextvars` vs `unbind_contextvars`:** structlog 25.x's `bind_contextvars` returns a Mapping of tokens for use with `reset_contextvars(**tokens)`. If the installed version doesn't expose this, fall back to `unbind_contextvars(*keys)` which removes by key name. The try/except handles both code paths defensively.
