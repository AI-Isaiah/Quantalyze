<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| untrusted client body → route | name + draft + (hostile) allocator_id cross here |
| route → Postgres (RLS) | user-scoped client; auth.uid() binds the write |
| route → client (error path) | DB error text must not leak schema/column names |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-23-05 | Spoofing/Tampering | POST/PUT body allocator_id | mitigate | allocator_id always = user.id from withAllocatorAuth; body field dropped; RLS WITH CHECK is defense-in-depth; test asserts the drop |
| T-23-06 | Tampering/DoS | malformed/oversized draft body | mitigate | scenarioDraftSchema on the wire + name 1..120; 400 before rate-limit; reasonable JSONB bound via the schema shape |
| T-23-07 | Information Disclosure | DB error envelope | mitigate | redact: console.error + captureToSentry server-side, stable UI-SPEC message to client (F5a/F5b); never echo error.message |
| T-23-08 | DoS | write flood from one allocator | mitigate | userActionLimiter per user.id, consumed AFTER validation (503 if misconfigured else 429 + Retry-After) |
| T-23-09 | Information Disclosure | caching of allocator payloads | mitigate | NO_STORE_HEADERS on every response (success + error) |
| T-23-10 | Information Disclosure | existence oracle on non-owned id | mitigate | non-owned id matches 0 rows → 404 (not 403); does not reveal existence |
</threat_model>
