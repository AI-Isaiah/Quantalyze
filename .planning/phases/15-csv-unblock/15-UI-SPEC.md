# Phase 15 — UI Design Contract

> Visual and interaction contract for the CSV branch of the strategy onboarding wizard. Generated by gsd-ui-researcher; validated by gsd-ui-checker before planner consumption.

This phase ships **net-new client code** on the existing strategy onboarding wizard (`/strategies/new/wizard`). The CSV branch reuses the existing `WizardChrome` shell, the existing `Button`/`Input`/`Card`/`Modal` primitives, and the project's existing token CSS variables. No new design system. No shadcn. Honor `DESIGN.md` as the visual contract; deviations are flagged in §11 below.

## 7. Interaction Contracts

### 7.1 Wizard CSV branch — happy path

```
/strategies/new/wizard?source=csv
   └─> Step 01 / 03 — Upload
       ├─ User picks format via segmented control (default: daily_returns)
       ├─ User drops file OR clicks "Choose file" (both paths → same handler)
       ├─ Client-side guards: file size ≤ 10 MB, extension == .csv
       ├─ On guard fail: render <CsvValidationEnvelope> inline above drop zone; do NOT advance
       └─> POST file + format to FastAPI /csv/validate (multipart/form-data)
           ├─ 200 OK + summary payload → advance to Preview
           └─ 4xx + envelope → render envelope; user fixes file; click "Try again"

   └─> Step 02 / 03 — Preview
       ├─ Card displays: row count, date range (earliest..latest), columns detected,
       │  first-3-rows + last-3-rows table
       ├─ Primary CTA "Submit strategy" enabled iff validation passed
       ├─ Ghost button "Back" returns to Upload (file state cleared so user must re-upload)
       └─> Click "Submit strategy" → advance to Submit

   └─> Step 03 / 03 — Submit
       ├─ Read-only summary (codename, format, row count, date range)
       ├─ Click "Submit strategy" → POST to FastAPI csv-finalize endpoint
       │  → on 200 OK: clear wizard state, router.push('/strategies/[id]?wizard_submitted=1')
       │  → on error: render envelope; preserve summary; user can retry without re-uploading
       └─ "Back" returns to Preview (file + validation state preserved)
```

### 7.2 Sub-step navigation rules

- **Forward navigation** is gated on validation success. "Submit strategy" on Preview is `disabled={!validationPassed}`.
- **Backward navigation** is always allowed and preserves uploaded-file state (mirrors `MetadataStep` ↔ `SubmitStep` back-and-forth in `WizardClient.tsx` lines 391–408).
- **Sidestep navigation** out of the wizard via `Delete draft` follows the existing `WizardChrome` modal-confirm flow unchanged.
- **`Request a Call`** ghost link in the footer remains visible across all 3 sub-steps (same as 4-step API branch).
- **Resume banner** logic in `WizardClient.tsx` (lines 87–89) extends to the CSV branch unchanged — a draft with `?source=csv` resumes at the last persisted sub-step.

### 7.3 Validation error rendering rules

All Phase 15 errors render through `<CsvValidationEnvelope>`. The envelope's visual contract:

```
┌────────────────────────────────────────────────────────────┐
│ [text-sm font-semibold text-negative]                      │
│ Title — e.g. "12 rows failed validation"                   │
│                                                            │
│ [text-xs text-text-secondary]                              │
│ Cause — e.g. "Across 3 rule categories: monotonic dates,   │
│ NAV non-zero, USD-or-blank currency."                      │
│                                                            │
│ [<details> · text-xs · 1 per rule category]                │
│ ▸ Monotonic dates (5 rows) [click to expand]               │
│   • Row 14: 2026-03-04 came after 2026-03-08               │
│   • Row 15: 2026-03-04 duplicated row 14                   │
│   ...                                                      │
│ ▸ NAV non-zero (4 rows) [click to expand]                  │
│ ▸ Currency must be USD or blank (3 rows) [click to expand] │
│                                                            │
│ [text-[11px] text-text-muted]                              │
│ correlation_id: — (Phase 15 placeholder; Phase 16 wires)   │
└────────────────────────────────────────────────────────────┘
```

**Background:** `border border-negative/30 bg-negative/5 rounded-md px-4 py-3` (exact match with `ConnectKeyStep` line 296 — `data-testid="wizard-connect-error"` precedent).

**Submit blocking rule:** ANY rule violation (max 10MB, monotonic dates, NAV non-zero, daily return > -100% impossible, daily Sharpe > 10 sentinel suspicious, trading-window check, USD-or-blank currency) blocks submit. There is no "warning + override" UX in v0 — Phase 17 may relax later. Keep `disabled={!validationPassed}` on the Submit CTA at all times.

### 7.4 Loading states

- **Upload submit (file → /csv/validate):** Primary CTA shows label "Validating…" + `disabled` (mirrors `ConnectKeyStep` line 328).
- **Submit step (Submit → csv-finalize):** Primary CTA shows label "Submitting…" + `disabled` (mirrors `SubmitStep` line 190).
- **No spinner needed for client-side parsing** — file under 10 MB parses in well under 100 ms; reserve loading copy for network-bound transitions only.
- **No skeleton on Preview** — Preview is rendered synchronously after the server's 200 OK; if the server is slow enough to need a skeleton, render the existing wizard `Saved · HH:MM` indicator and let the user wait. This matches `SyncPreviewStep`'s elapsed-counter pattern (lines 423–460) at a smaller scale.

### 7.5 Success state

On Submit success, redirect to `/strategies/[id]?wizard_submitted=1` (verbatim match with `WizardClient.tsx` line 218). The destination page is responsible for rendering the "Submitted for review" treatment via `<FactsheetPreview verificationState="pending" />` — Phase 15 does not touch the post-submit surface.

The `<TrustTierLabel trustTier="csv_uploaded" />` MUST be visible on `/strategies/[id]` immediately after redirect — confirms data wiring landed.

### 8.7 Error state copy

All errors flow through `<CsvValidationEnvelope>`. The envelope's title + summary copy is computed from the pandera error array. Copy templates:

| Scenario | Title | Summary |
|----------|-------|---------|
| File >10 MB | `File too large` | `Maximum file size is 10 MB. Your file is <X> MB. Trim it or split it before retrying.` |
| Wrong extension | `That is not a CSV` | `Only .csv files are accepted. Convert your file and try again.` |
| Empty / no data rows | `No data found` | `Your file has a header but no data rows. Double-check the export.` |
| Format / parse error | `We could not read your file` | `<n> rows could not be parsed. Most often this is a missing column or stray quote character.` |
| Pandera validation (1 rule) | `<n> row<s> failed validation` | `Rule violated: <rule_name_human>. Expand below for the row-level breakdown.` |
| Pandera validation (>1 rule) | `<n> rows failed validation` | `Across <r> rule categories: <comma-joined list>. Expand each category below for row-level details.` |

| Server 5xx | `Validation service unavailable` | `Our validation service hit an error. The team has been notified. Retry shortly, or click Stuck? Request a Call below.` |
| Submit-step finalize fail | `Submission failed` | `Your file validated cleanly, but saving the strategy hit an error. Click Submit strategy again to retry — your data is unchanged.` |
