# Phase 09.1 Plan 03: TileConfig {k,w} shape + LAYOUT_VERSION 4 + V2 hook split Summary

Type-system foundation for the v4 grid: TileConfig narrowed to `{k, w: 1|2|3|4}`, LAYOUT_VERSION bumped 3â†’4 with the designer's 7-entry Overview default, and useDashboardConfig split into a legacy hook (frozen v3) plus `useDashboardConfigV2` (v4) sharing a single localStorage key per D-02.

## Context

This plan ships the type contract that Plans 05 (WidgetGrid + AllocationDashboardV2) and 11 (Tweaks panel) consume. The shape change from `{i, widgetId, x, y, w, h}` (react-grid-layout grid units) to `{k, w}` (4-col CSS-grid spans) is the structural prerequisite for the entire v2 dashboard body. The LAYOUT_VERSION bump 3â†’4 forces every existing v3 layout to reset to the new 7-entry default on next V2 load â€” Voice-D8 accepted precedent (no banner, custom layouts lost once).

Crucially the bake-window approach requires BOTH hooks to coexist on the SAME `quantalyze-dashboard-config` localStorage key (D-02). Splitting the key would create orphan parallel state when an allocator flips the `allocations.ui_v2` flag mid-session â€” instead, each hook's loadConfig path resets to its own default layout when the persisted `layoutVersion` doesn't match what it knows about. The 1â†’2 (Phase 05) and 2â†’3 (Phase 08) bumps both used this pattern.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 â€” Blocking issue] WidgetMeta.defaultW kept as `number` instead of `1 | 2 | 3 | 4`**

- **Found during:** Task 1 typecheck (after writing types.ts + dashboard-defaults.ts).
- **Issue:** The plan's Task 1 action narrows `WidgetMeta.defaultW` from `number` to `1 | 2 | 3 | 4`. The existing `src/app/(dashboard)/allocations/lib/widget-registry.ts` has 50+ entries with values `3`, `4`, `6`, `12` â€” all of which become type errors under the narrowed `defaultW`. The plan explicitly says **"Do NOT edit widget-registry.ts in this plan"** (Plan 05 owns the registry overhaul). The two constraints are mutually exclusive: narrowing the type while not editing the registry leaves Task 2's typecheck-clean done-criterion unsatisfiable.
- **Fix:** Keep `WidgetMeta.defaultW: number` with a transition-note comment that explains the legacy/V2 split and points at Plan 05 for the future narrowing. Compensate at the runtime layer: the V2 hook's `clampWidth()` helper coerces registry values to 1..4 at `addWidget` time so persisted `TileConfig.w` is always a valid `1 | 2 | 3 | 4`. `defaultH` made optional since Plan 05 will drop it entirely.
- **Files modified:** `src/app/(dashboard)/allocations/lib/types.ts` (Task 1 + Task 2 commit).
- **Commit:** `c26c09a` (initial), `cfe4258` (clampWidth + comment + defaultH optional).

**2. [Rule 3 â€” Blocking issue] SSR-safety test rewritten to throw from `localStorage.getItem` instead of stubbing `window`**

- **Found during:** Task 3 first vitest run (6 V2 cases failing post-SSR-test with `Cannot read properties of undefined (reading 'event')` from React-DOM internals).
- **Issue:** The plan's Task 3 case 9 says "SSR safety: hook does not throw when `window` is undefined (use `vi.stubGlobal` to simulate)." Globally stubbing `window` to undefined while jsdom is active breaks React-DOM â€” React-DOM caches the real window reference at module load and crashes on the next `renderHook` even after the stub is restored.
- **Fix:** Test the same fallback path by making `localStorageMock.getItem` throw (simulating Safari private mode / quota errors / any hostile-storage environment). The hook's `loadV2Config` try/catch swallows the error and returns DEFAULT_LAYOUT â€” same code path the SSR check goes down via the `typeof window === "undefined"` early return. Restore `getItem` at the end of the test.
- **Files modified:** `src/app/(dashboard)/allocations/hooks/useDashboardConfig.test.ts`.
- **Commit:** `37033aa`.
