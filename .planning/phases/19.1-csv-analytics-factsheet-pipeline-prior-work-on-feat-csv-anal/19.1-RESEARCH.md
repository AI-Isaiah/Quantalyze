### Pattern 1: Migration Self-Verifying DO Block

**What:** Every additive migration ends with a `DO $$ ... END $$` block that asserts the
expected state (table exists, RLS enabled, N policies present, RPC registered). Failure
raises `RAISE EXCEPTION` and rolls back the whole transaction.

**When to use:** Every migration in this phase (both `csv_daily_returns.sql` and
`compute_analytics_from_csv_kind.sql`).

**Example (from `supabase/migrations/20260501055202_strategy_verifications.sql:296-370` — migration 093):**

```sql

DO $$
DECLARE
  v_column_count INT;
  v_rls_enabled  BOOLEAN;
  v_policy_count INT;
  v_index_count  INT;
  v_fn_oid       OID;
BEGIN
```

### Anti-Patterns to Avoid

- **Re-introducing the explicit `csv_daily_returns_strategy_date_idx` index:** The UNIQUE
  constraint on `(strategy_id, date)` creates an implicit B-tree that already serves the
  worker SELECT + ON CONFLICT upsert. Explicit index doubles write I/O for zero planner
  benefit. PR #272 dropped it; do not re-add (migration-reviewer will flag again).
- **Splitting missing-strategy and wrong-owner into two distinct error codes:** Authenticated
  callers can enumerate which `strategy_id` UUIDs exist by passing `[]` as `p_rows` and
  reading the error-code distinction. The collapsed `42501` "not accessible" branch closes
  the probe-oracle (PR #272). Do not re-split.
- **Reading `dailyReturnsSeries` via closure capture in `unifiedCsvFinalizeHandler`:**
  Code-review caught this — pass explicitly through the function signature so the type
  system enforces the dependency. (Task 10 CONTEXT.md decision.)
- **Marking the strategy `analytics-missing` row stuck in `computing` without log evidence:**
  PR #273 wraps `_mark_unrecoverable` in try/except but logs a warning before re-raising —
  necessary so a stuck row can be located in logs.
- **Using `next/cache`'s `unstable_after`:** The project's established import is
  `import { after } from "next/server"`. Do not switch to a different `after()` import
  source.
- **`vi.mock("next/server")` shape mismatch:** A naive shim that calls `setImmediate(fn)`
  flushes only one microtask tick and races against nested awaits inside the after()
  callback. Use the Promise-storing variant from `finalize-wizard/route.test.ts:234-242`.
- **`SET ROLE service_role`-style narrowing on the GRANT:** Pseudonym in code review for
  the same probe-oracle root cause; `service_role` strips `auth.uid()` to NULL and
  triggers 42501 on every legitimate call.
- **Reading `process.env.USE_COMPUTE_JOBS_QUEUE` outside `after()`:** The flag gates the
  enqueue side effect, not the rest of the route. Read it inside the `after()` body
  (mirrors `finalize-wizard/route.ts:619-622`).
- **Removing `analyticsMissingMessage.ts` BEFORE Task 16 verifies the pipeline ships
  metrics in production:** Rule 6 (root-cause obsession) — the stop-gap was installed
  because the pipeline was missing; the removal proves the pipeline works. Reversing
  that order ships broken UI to LPs.

## Common Pitfalls

### Pitfall 1: Migration filename collision with an unmerged remote schema_migrations row

**What goes wrong:** Phase 19 PR-Y1 discovered that the remote `supabase_migrations.schema_migrations`
table has 32 timestamped rows from `20260424012820` to `20260515095804` whose SQL bodies
were applied via MCP `apply_migration` but whose timestamp does NOT match a local file.
Adding `20260522*` migrations is safe (no overlap with that block), but a subagent who
copies a different timestamp pattern could accidentally collide.

**Why it happens:** MCP `apply_migration` writes the timestamp it generates; local files
were renamed lazily.

**How to avoid:** Use timestamps strictly `> 20260521210000` (latest on main). PR #270's
`20260522120000` and `20260522120100` are safe and reusable. Run `ls supabase/migrations/
| tail -3` before naming.

**Warning signs:** `supabase db push --include-all --dry-run` reports "Found local
migration ... that is below remote tip" or "Remote migration versions not found in local."

### Pitfall 2: `vi.mock("next/server")` setImmediate vs Promise.resolve().then() race

**What goes wrong:** A naive `after = (fn) => setImmediate(fn)` shim flushes the
microtask queue exactly once. If the after() callback awaits an async DB call (which the
enqueue does), the await resolves on the next microtask tick — too late for the test's
assertion phase.

**Why it happens:** Vitest's node env doesn't have the Vercel function lifecycle; the
shim has to replicate Vercel's after-response semantics inside the same Promise chain
the test awaits.

**How to avoid:** Use the Promise-storing variant from
`finalize-wizard/route.test.ts:234-242` that stores the `Promise.resolve().then(fn)` so
`flushAfter()` can `await` it deterministically.

**Warning signs:** Test passes locally but fails in CI; or the assertion about
`enqueue_compute_job` being called fires before the after() callback completes.

### Pitfall 3: `complete_with_warnings` typing the React tree before the DB writes it

**What goes wrong:** Task 7 extends the TS union with `"complete_with_warnings"`. If
the analytics-runner promotion is NOT live (Task 14 — Railway deploy), some prod rows
will still be `"complete"` or `"complete_with_warnings"` from the existing
`run_strategy_analytics` (which already writes `complete_with_warnings` when
`used_heuristic_capital` fires). The TS surface must handle both states gracefully.

**Why it happens:** `complete_with_warnings` ALREADY exists in the DB (verified in
`analytics-service/services/analytics_runner.py:1320-1330` — quoting the long comment
block above the consumer_specific_flags check). The TS union just doesn't know about it.

**How to avoid:** Extending the union is a non-breaking change for consumers that only
check `=== "complete"` — they correctly route to the loading/error path for the new
state and don't crash. Any consumer that wants to surface warnings (like the existing
`SyncProgress.tsx` switch) must add a branch.

**Warning signs:** `tsc --noEmit` flags exhaustive switches that don't include
`complete_with_warnings`; `assertNever` raises at runtime for an unhandled DB value.

### Pitfall 4: Skipping live-DB tests silently on CI

**What goes wrong:** Live-DB tests gated on `TEST_SUPABASE_DB_URL` skip when the env
var is unset. If the gate-check is `pytest.mark.skipif(reason="…")` without a
`reason=` string the operator can read, the skip looks like "test ran and passed" in
the CI output.

**Why it happens:** Rule 12 violation (Fail loud).

**How to avoid:** Include a verbose `reason=` string in every `skipif` (existing
pattern in `test_resend_correlation_rls.py:33-35`). For Task 12, also surface the
skip count in the phase-completion checklist so a reviewer can verify the live tests
were intentionally skipped vs accidentally never run.

**Warning signs:** Phase 19.1 ships and the live-DB tests have `0` runs across all CI
artifacts — that's a silent-skip, not a clean pass.

### Pitfall 5: `analyticsMissingMessage` deletion ordering

**What goes wrong:** Removing `analyticsMissingMessage.ts` before the pipeline is live
in prod (Task 14 + Task 15 + Task 16 verified) means CSV strategies render the standard
"Analytics are being computed. Check back soon." message indefinitely (no worker job
runs against the un-deployed handler; or the enqueue never fires because the flag is
still off).

**Why it happens:** Convenient task-order optimization — "delete the file while
typing types changes" — that bypasses the actual gating dependency.

**How to avoid:** Task 17 is explicitly the LAST task in the phase. Plan-checker
should reject any plan that places Task 17 before Task 16's verification step.

**Warning signs:** Plan reviewer sees Task 17 in any wave other than the final.

### Pitfall 6: Reading the `database.types.ts` Functions union for the new RPC

**What goes wrong:** The generated `database.types.ts` file does not yet know about
`persist_csv_daily_returns` (or `finalize_csv_strategy`, per the existing comment at
`csv-finalize/route.ts:338-340`). Calling `supabase.rpc("persist_csv_daily_returns",
...)` fails TS without a cast.

**Why it happens:** Type regeneration is out-of-band (`supabase gen types` runs against
the test project; the file is checked in manually after migration apply).

**How to avoid:** Cast through `unknown` mirroring the existing `finalize_csv_strategy`
pattern at `csv-finalize/route.ts:341-356`. Comment with the same C-0155/C-0157
reference (audit identifier — the cast is acceptable as long as the call shape matches
the migration RPC signature exactly).

**Warning signs:** `npx tsc --noEmit` fails with "Argument of type
'\"persist_csv_daily_returns\"' is not assignable to parameter of type 'never'."

### Pitfall 7: Watchdog reclaiming jobs because handler timeout >= watchdog threshold

**What goes wrong:** Forgetting the `WATCHDOG_PER_KIND_OVERRIDES` entry causes the
global 10-min default to apply. Handler timeout is 10 min. `test_every_kind_has_watchdog_headroom`
asserts watchdog > handler — strictly greater — so the test fails CI. Worse, if it
somehow ships, the watchdog reclaims still-running 9-min CSV computations and the
wizard hangs.

**Why it happens:** Two-file change (handler timeout in `services/job_worker.py`, watchdog
override in `main_worker.py`). Easy to update one and forget the other.

**How to avoid:** Tasks 5 and 6 must be a single atomic plan slice — both files touched
together. Plan-checker should flag any plan that splits them.

**Warning signs:** `test_every_kind_has_watchdog_headroom` fails with "Kind
'compute_analytics_from_csv': handler timeout 10.0m exceeds watchdog threshold 10m".

### Pitfall 8: Anonymous SELECT on csv_daily_returns leaking row data

**What goes wrong:** The 3-tier RLS includes service_role ALL + owner SELECT + admin
SELECT — no anon policy. The anon path is denied. BUT a sibling kind that publishes
the `sparkline_returns` series via `upsert_strategy_analytics_series_batch` MIGHT
expose row-derivable data. Quantalyze publishes `daily_returns_grid` for published
strategies via `fetch_strategy_lazy_metrics`.

**Why it happens:** Aggregated published artifacts (sparkline / monthly grid) live in
`strategy_analytics_series`, not in `csv_daily_returns`. If the strategy is published,
the aggregated chart-feeding data IS public — that's the existing model for API-backed

### `parseDailyReturnsSeries()` — route boundary validator with duplicate-date guard

```typescript
// Source: PR #274 git show 17705685:src/app/api/strategies/csv-finalize/route.ts
// (canonical version after PR #274 added the duplicate-date guard)

const MAX_DAILY_RETURNS_ROWS = 5000;
const DATE_REGEX = /^\d{4}-\d{2}-\d{2}$/;

export interface CsvDailyReturnRow {
  date: string;       // YYYY-MM-DD
  daily_return: number;
}

function parseDailyReturnsSeries(raw: unknown): {
  ok: true; rows: CsvDailyReturnRow[];
} | {
  ok: false; code: string; message: string; debug_context?: Record<string, unknown>;
} {
  if (!Array.isArray(raw)) {
    return { ok: false, code: "CSV_INVALID_FORMAT", message: "daily_returns_series must be an array" };
  }
  if (raw.length > MAX_DAILY_RETURNS_ROWS) {
    return {
      ok: false,
      code: "CSV_INVALID_FORMAT",
      message: `daily_returns_series exceeds ${MAX_DAILY_RETURNS_ROWS} rows (got ${raw.length})`,
      debug_context: { length: raw.length },
    };
  }
  const seen = new Set<string>();
  const rows: CsvDailyReturnRow[] = [];
  for (let i = 0; i < raw.length; i++) {
    const r = raw[i] as Record<string, unknown> | null;
    if (!r || typeof r !== "object") {
      return { ok: false, code: "CSV_INVALID_FORMAT", message: `row ${i} is not an object` };
    }
    if (typeof r.date !== "string" || !DATE_REGEX.test(r.date)) {
      return {
        ok: false,
        code: "CSV_INVALID_FORMAT",
        message: `row ${i}.date must be YYYY-MM-DD`,
        debug_context: { row: i },
      };
    }
    if (typeof r.daily_return !== "number" || !Number.isFinite(r.daily_return)) {
      return {
        ok: false,
        code: "CSV_INVALID_FORMAT",
        message: `row ${i}.daily_return must be a finite number`,
        debug_context: { row: i },
      };
    }
    if (seen.has(r.date)) {                                            // PR #274 dup-date guard
      return {
        ok: false,
        code: "CSV_INVALID_FORMAT",
        message: `duplicate date at row ${i}: ${r.date}`,
        debug_context: { row: i, date: r.date },
      };
    }
    seen.add(r.date);
    rows.push({ date: r.date, daily_return: r.daily_return });
  }
  return { ok: true, rows };
}
```

### `run_csv_strategy_analytics` (Python) — final hardened state

```python
# Source: PR #273 git show 01cbea60:analytics-service/services/analytics_runner.py:1446-1555
async def run_csv_strategy_analytics(strategy_id: str) -> dict:
    """CSV → analytics pipeline. Loads csv_daily_returns, builds pd.Series,
    calls compute_all_metrics. No position reconstruction, fills, trade_mix,
    or volume metrics (CSV has none of those — those panels stay null).

    Sets computation_status='computing' on entry; 'complete' or 'failed' on exit.
    """
    supabase = get_supabase()

    def _mark_computing() -> None:
        supabase.table("strategy_analytics").upsert(
            {"strategy_id": strategy_id, "computation_status": "computing"},
            on_conflict="strategy_id",
        ).execute()
    await db_execute(_mark_computing)

    try:
        def _load_series():
            return (
                supabase.table("csv_daily_returns")
                .select("date, daily_return")
                .eq("strategy_id", strategy_id)
                .order("date")
                .execute()
            )
        rows_resp = await db_execute(_load_series)
        data = rows_resp.data or []

        if len(data) < 2:
            def _mark_failed():
                supabase.table("strategy_analytics").upsert(
                    {
                        "strategy_id": strategy_id,
                        "computation_status": "failed",
                        "computation_error": "Insufficient CSV history. At least 2 data points required.",
                    },
                    on_conflict="strategy_id",
                ).execute()
            await db_execute(_mark_failed)
            raise HTTPException(status_code=400, detail="Insufficient CSV history")

        import pandas as pd
        dates = pd.DatetimeIndex([r["date"] for r in data])
        values = [float(r["daily_return"]) for r in data]
        returns = pd.Series(values, index=dates, name="returns")

        benchmark_rets, benchmark_stale = None, True
        try:
            benchmark_rets, benchmark_stale = await get_benchmark_returns("BTC")
        except Exception as exc:  # noqa: BLE001
            logger.warning("csv analytics: benchmark fetch failed for %s: %s", strategy_id, exc)

        metrics_result = compute_all_metrics(returns, benchmark_rets)

        data_quality_flags: dict = {"csv_source": True}
        if benchmark_stale or benchmark_rets is None:
            data_quality_flags["benchmark_unavailable"] = True
            data_quality_flags["benchmark_note"] = "Benchmark data unavailable."

        def _mark_complete():
            payload = {
                "strategy_id": strategy_id,
                "computation_status": "complete",
                "computation_error": None,
                "data_quality_flags": data_quality_flags,
                "trade_metrics": None,
                "volume_metrics": None,
                "exposure_metrics": None,
            }
            payload.update(metrics_result.metrics_json)
            supabase.table("strategy_analytics").upsert(
                payload, on_conflict="strategy_id"
            ).execute()
        await db_execute(_mark_complete)

        if metrics_result.sibling_kinds:
            def _upsert_siblings():
                supabase.rpc(
                    "upsert_strategy_analytics_series_batch",
                    {"p_strategy_id": strategy_id, "p_kinds": metrics_result.sibling_kinds},
                ).execute()
            await db_execute(_upsert_siblings)

        return {"status": "complete", "strategy_id": strategy_id}

    except HTTPException:
        raise
    except Exception as exc:  # noqa: BLE001
        logger.error("csv analytics failed for %s: %s", strategy_id, exc)
        def _mark_unrecoverable():
            supabase.table("strategy_analytics").upsert(
                {
                    "strategy_id": strategy_id,
                    "computation_status": "failed",
                    "computation_error": "CSV analytics computation failed.",
                },
                on_conflict="strategy_id",
            ).execute()
        try:
            await db_execute(_mark_unrecoverable)
        except Exception as mark_exc:  # noqa: BLE001
            # PR #273 — log before re-raise so a stuck `computing` row has log evidence
            logger.warning(
                "csv analytics: could not mark strategy %s unrecoverable: %s",
                strategy_id,
                mark_exc,
            )
        raise HTTPException(status_code=500, detail="CSV analytics computation failed")
```
