## Rollback Path

Recovery procedure if the production apply lands in a partial / inconsistent state. The migrations are designed so this should not happen — but record it in advance per Rule 12 + Rule 10.

**Case A — Migration 1 (`20260522120000_csv_daily_returns`) applied, Migration 2 (`20260522120100_compute_analytics_from_csv_kind`) failed:**
- Migration 1 is purely additive (CREATE TABLE + CREATE FUNCTION + GRANT). The table is empty post-apply; the RPC is unreachable from production callers until Plan 08 flips `USE_COMPUTE_JOBS_QUEUE=true`. Safe to keep.
- Retry Migration 2 once root cause is identified. If retry is not possible, drop the additions manually:
  ```sql
  -- Only if Migration 2 is permanently abandoned:
  DROP FUNCTION IF EXISTS public.persist_csv_daily_returns(uuid, uuid, jsonb);
  DROP TABLE IF EXISTS public.csv_daily_returns;
  -- And delete the row from supabase_migrations.schema_migrations:
  DELETE FROM supabase_migrations.schema_migrations WHERE version = '20260522120000';
  ```

**Case B — Migration 2 applied, Migration 1 failed:**
- Topologically impossible because Migrations are applied in timestamp order in Task 2. If a manual MCP error sequenced them backwards (e.g. operator ran Migration 2 first by mistake), Migration 2 would fail because it references the `compute_job_kinds` registry that Migration 1 does NOT populate — but Migration 1 was never the only writer of `compute_job_kinds`, so Migration 2 could actually succeed standalone. In that case the `compute_jobs_kind_target_coherence` CHECK admits the new kind without the table existing yet. Safe state (table absence will surface as a runtime FK / SELECT error in the worker, NOT a constraint violation).
- Recovery: apply Migration 1 to fill the missing pieces. No manual DDL required.

**Case C — Constraint swap in Migration 2 dropped the OLD constraint but the new CHECK fails to install (e.g. self-verify block raised mid-transaction):**
- The DROP+ADD in Migration 2 runs in a single transaction. If the new CHECK self-verify fires, the entire transaction rolls back — production is left with the old `compute_jobs_kind_target_coherence` constraint intact. No manual rollback needed.
- If somehow the DDL transaction landed without the new CHECK (e.g. operator interrupted mid-stream via MCP cancel + the platform applied partial DDL), manually restore the old shape:
  ```sql
  ALTER TABLE compute_jobs DROP CONSTRAINT IF EXISTS compute_jobs_kind_target_coherence;
  -- Re-add the OLD shape (from migration 20260510175507_process_key_long_compute_job_kinds_repair.sql);
  -- copy the old CHECK clause body verbatim from that file:
  ALTER TABLE compute_jobs ADD CONSTRAINT compute_jobs_kind_target_coherence
    CHECK (<OLD CHECK CLAUSE BODY FROM 20260510175507>);
  ```
- Reference for the old clause body: `supabase/migrations/20260510175507_process_key_long_compute_job_kinds_repair.sql`. Planner note: if a `supabase/migrations/down/` directory exists with a paired down-migration, prefer using that; otherwise the manual restore above is the documented path. The down-migration is not present at planning time; record this in SUMMARY if encountered.

**Reach for rollback only if:** Task 4 schema-parity diff cannot be reconciled within the planning session AND production traffic is impacted. For non-impacting diffs (e.g. a probe payload-shape difference because of a transient Supabase MCP response variation), prefer a retry of Task 3 over a rollback.
