## Decisions Made

- **psycopg connection-pattern reuse over TS @/lib/test-helpers/live-db.ts:** The RPC under test is SQL-side and several assertions inspect `pg_indexes`, `pg_proc`, and `current_user`. psycopg gives direct access to those system catalogs and to per-transaction `SET LOCAL ROLE`, which the Supabase TS client does not.
- **Test 6 split into two functions:** Pinning both halves of T-19.1-02 (positive: authenticated executes; negative: service_role without jwt claims → 42501) as separate tests makes the GRANT-shape contract more legible than a single test with two phases. If a future migration accidentally narrows the GRANT, the positive test fails first with a clean diagnostic.
- **Tests 9-11 invoke the runner in-process:** `asyncio.run(run_csv_strategy_analytics(sid))` is the closest in-test simulation of the worker pipeline against the live DB. Inserting a `compute_jobs` row and polling for the Railway worker would couple the test to whether a worker instance is currently polling qmnijlgmdhviwzwfyzlc, which is a flaky external dependency.
- **Test 9 expects EXACTLY `failed` for 1-row:** `analytics_runner.py:1490` raises `HTTPException(400, "Insufficient CSV history")` after marking the row `failed`. The plan's behavior text accepts EITHER `complete_with_warnings` OR `failed`; the actual contract is the latter, so the test pins the precise observed behavior to prevent future drift toward a more lenient `complete_with_warnings` path that would mask the data-quality problem.
- **Test 11 NaN-Inf accepts `complete` / `complete_with_warnings` outcomes too:** pandas may drop NaN entries before `compute_all_metrics` runs, producing a sanitized payload that the runner marks `complete`. Both outcomes (sanitized-success OR structured-failure) satisfy the anti-poisoning contract; what's NOT allowed is a stuck `computing` row OR a Python traceback in `computation_error`.
